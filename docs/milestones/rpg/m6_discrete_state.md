# M6 — Discrete State: Menus, UI, Cuts, and Game-State Conditioning

**Objective:** handle the non-physics half of RPGs — menus, dialogs, map/fast-travel,
death/respawn, loading — and persistent scalar state (HP/stamina/runes), by conditioning the world
model on segment types and a structured-state vector. This milestone also draws MIRA's first
explicit **mechanics/state boundary**: an executable mechanics graph for one narrow loop whose
state deltas condition the model.

## 0. Context & prerequisites

- **Depends on:** M4 checkpoint (can run parallel to M5). Consumes M2's segment annotations
  (menu/loading/cutscene/death events) and Tier-1 HUD state extraction.
- M4 deliberately excluded menu spans from training; M6 brings them back in as *conditioned* data.

## 1. Design

### 1.1 Segment-type conditioning

- Per-latent-frame segment token: `{gameplay, menu, cutscene, loading, death}` from M2 events,
  embedded and added to the action-conditioning stream (same injection point as the
  `ActionEncoder` output; a small `SegmentEncoder` alongside it).
- At inference the segment stream is an **input** — during free play it comes from the mechanics
  layer (menu open/close is an explicit state machine), never from the model's own guess. This is
  the core "conditioned, not hallucinated" contract.
- Training: include menu/cutscene spans; drop the segment token CFG-style so the model degrades
  gracefully when the signal is absent.

### 1.2 Persistent-state channel

- Structured-state vector per latent frame: normalized HP/FP/stamina, currency (symlog), time-of-
  day, inventory hash (bucketed/embedded), lock-on flag — sourced from M2 Tier-1/2 taps.
- Encoding: small MLP → conditioning token per frame; **NaN = unknown per element**, masked to
  learned tokens (extend the M1 presence-masking pattern; sparse/missing state is the norm).
- Evaluate whether conditioning on state makes the *rendered HUD* coherent with it over rollouts.

### 1.3 Mechanics/state ownership doc (a deliverable, not an afterthought)

A written contract, kept in `docs/`, defining for each system whether it is **explicit**
(health, stamina, inventory, quest flags, collision authority, save/load, NPC/AI decisions,
menu state machine) or **neural** (visual continuation, animation interpolation,
lighting/material detail, uncertain residual dynamics), plus the interface between them
(state deltas → conditioning tokens). This doc is the seed of the hybrid runtime (M8) and the
remaster track's R3.

### 1.4 First executable mechanics graph (narrow loop)

- Pick **one** loop; recommended: stamina-gated dodge (input → stamina check → dodge state →
  stamina delta → regen), or menu open/close. Small, typed, deterministic transition system in a
  new `src/mira/mechanics/` package: nodes with reads/writes, conditions, effects, timers.
- Trained/validated against traces: replay captured inputs through the graph and compare its
  state trajectory to HUD-extracted ground truth.
- Integration: graph emits per-frame state deltas → §1.2 state vector → the model renders. The
  model never invents stamina values in this loop.

### 1.5 Metrics

- **Mode-switch fidelity:** same context, segment token flips gameplay→menu — the rollout must
  produce the correct discrete transition (menu-frame classifier from M2's detector); and must
  NOT switch when the token doesn't flip (false-positive rate).
- **HUD-value coherence:** OCR the generated HUD over 30s rollouts; compare against the
  conditioning state trajectory (mean absolute error on bar fractions, digit accuracy on
  counters).
- Regression: M4 controllability metrics must not degrade with the new conditioning enabled.

## 2. Phased execution

1. **Phase A — Segment conditioning.** SegmentEncoder + training with menu data reinstated;
   mode-switch metric; iterate to threshold.
2. **Phase B — State channel.** State vector encode path + HUD-coherence metric; ablate with/
   without state conditioning.
3. **Phase C — Mechanics graph.** `src/mira/mechanics/` prototype + trace-replay validation +
   end-to-end demo: scripted input → graph → state deltas → conditioned rollout where the
   rendered stamina bar tracks the graph's stamina.
4. **Phase D — Decision & docs.** Decide (from Phase A/B results) whether menus are generated by
   the same model or delegated to a separate branch/renderer; write the ownership doc; final
   report.

## 3. Non-goals

- No full quest/inventory simulator — exactly one narrow loop (breadth is R3/M8 territory).
- No long-horizon memory interaction (M5 merges with this in M7).
- No UI compositing engine; menus stay generated (unless Phase D decides otherwise — that
  decision is a deliverable, not presumed).

## 4. Tests & verification

- SegmentEncoder / state-encoder unit tests incl. all-NaN masking and CFG dropout.
- Mechanics graph: unit tests per rule; property test (stamina never negative, regen bounded);
  trace-replay test against HUD ground truth with tolerance.
- Metric unit tests with synthetic fixtures; Hydra smoke test for new config blocks (additive,
  default-off ⇒ M4/M5 configs unaffected; checkpoint-compat per `REMOVED_CONFIG_FIELDS`
  discipline).
- `pixi run verify` green.

## 5. Risks & mitigations

| Risk | Mitigation |
|---|---|
| HUD OCR ground truth too noisy to train/eval against | Confidence-filtered labels; evaluate metric noise floor on real footage first |
| Menu frames poison gameplay dynamics (shared weights) | Segment conditioning + CFG dropout; ablate a separate-branch decoder in Phase D if leakage is measured |
| State vector conditioning gets ignored (like actions in M4) | Reuse M4's swap-divergence diagnostic on state: perturb HP, check rendered HUD diverges |
| Mechanics graph scope creep | One loop, hard-scoped; anything else is a backlog note for M8/R3 |
| Annotation gaps (unlabeled cutscenes etc.) | Segment-token dropout at train time makes missing labels a soft failure |

## 6. Sequencing & effort

A (~2 weeks) → B (~2 weeks) → C (~2 weeks) → D (~1 week). Parallelizable with M5 (different
model-surface areas; coordinate on config schema).

**Definition of done:** mode-switch and HUD-coherence metrics merged, thresholds met over 30s
rollouts; M4 metrics non-regressed; one mechanics graph validated against traces and driving a
conditioned rollout end-to-end; ownership doc published; menus-same-model-or-branch decision
documented with evidence; `pixi run verify` green.
